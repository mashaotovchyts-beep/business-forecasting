# business-forecasting
1. Назва проекту / задачі
«Прогнозування загального товарообігу міжнародного маркетплейсу на 18 місяців наперед».

2. Тип моделі
Модель аналізу часових рядів (наприклад, бібліотека Prophet від Meta або модель ARIMA), а також регресійні моделі для врахування довгострокових трендів та сезонності продажів.

3. Посилання на таски в Jira / Trello / іншу систему
   [https://github.com/users/mashaotovchyts-beep/projects/1/views/1].

4. Дата запуску проекту / задачі в продакшн
Планова дата інтеграції алгоритму в робочі процеси фінансового департаменту — 16 червня 2026 року.

5. Розміщення проекту / задачі

Репозиторій: [https://github.com/mashaotovchyts-beep/business-forecasting].

Джерела вхідних даних: Внутрішня база даних продажів маркетплейсу. Дані беруться у вигляді історичних таблиць, що містять поля: date (дата), category (категорія товару), revenue_retail (роздрібна виручка) та margin (маржа).

Результати: Результати прогнозу автоматично вивантажуються в аналітичну базу даних (наприклад, PostgreSQL / BigQuery) у форматі структурованої таблиці з прогнозними значеннями виручки на наступні 18 місяців.

Передача даних у інші системи: Дані з бази автоматично експортуються в BI-систему (наприклад, Tableau або Power BI) для візуалізації фінансових звітів.

6. Регулярність запуску
Процес виконується регулярно раз на місяць (1-го числа кожного місяця), оскільки прогноз є довгостроковим і розраховується на 18 місяців наперед.

7. Замовник
Фінансовий департамент маркетплейсу (Financial Department).

8. Розробник(и)
Маркетолог-аналітик / Бізнес-аналітик Марія Отовчиць.

9. Яка проблема вирішується
Висока невизначеність щодо майбутніх фінансових потоків маркетплейсу. Без точного прогнозу товарообігу Фінансовий департамент не може ефективно планувати бюджети на маркетинг, логістику, закупівлі та розширення складських потужностей на наступні 1.5 роки, що створює ризики касових розривів.

10. Бізнес-вимоги до проекту
Алгоритм повинен автоматично розраховувати прогноз товарообігу (роздрібної виручки та маржі) з розбивкою по місяцях та ключових категоріях товарів. Готовий прогноз має автоматично завантажуватись в BI-систему для щомісячного фінансового та стратегічного планування компанії.

11. Метрики якості
Точність прогнозу оцінюється за метрикою MAPE. Цільовий показник: помилка прогнозу не повинна перевищувати 5-7% для стабільних категорій товарів.

12. Виключення з прогнозу / обмеження
Не прогнозуються продажі в розрізі окремих товарів (SKU), оцінюється лише загальний товарообіг мережі та великі категорії. Алгоритм не враховує форс-мажорні глобальні фактори (пандемії, повні блекаути чи блокування ключових логістичних шляхів).

13. Особливості / обмеження
Модель повинна обов'язково враховувати високу сезонність ритейлу (пікові продажі під час Чорної п'ятниці, Киберпонеділка та Різдвяних розпродажів), а також заплановані масштабні маркетингові кампанії маркетплейсу.

14. Економічна ефективність

Зниження ризиків касових розривів на 15% за рахунок точнішого планування витрат.

Оптимізація маркетингових бюджетів завдяки розумінню періодів спаду та росту продажів.

Економія часу співробітників фінансового відділу на ручне складання прогнозів.

16. Детальний опис проекту

Функціональні вимоги: Система повинна автоматично підключатися до джерела даних, очищати історію продажів від випадкових аномалій (сплесків/дефіцитів), будувати лінію тренду та сезонності на 18 місяців наперед і розраховувати верхню та нижню межу прогнозу.

Flow роботи: Завантаження історичних даних ➔ Очищення та агрегація даних за місяцями ➔ Запуск прогнозної моделі часових рядів ➔ Генерація прогнозу на 18 місяців ➔ Експорт результатів у BI-дашборд.

Блок-схема: Процес є лінійним: [Вхідні дані: Виручка та Маржа] ➔ [Модуль очищення даних від шумів] ➔ [Модель прогнозування (Prophet/ARIMA)] ➔ [База даних результатів] ➔ [BI-Дашборд фінансового директора].

17. Вимоги до даних

Структура: Таблиця повинна містити поля: date (формат YYYY-MM-DD), category_id (ідентифікатор категорії), revenue_retail (числове значення виручки), margin (числове значення маржі у валюті або %).

Обмеження щодо оновлення: Дані для моделі мають бути повністю сформовані та доступні не пізніше 2-го числа кожного звітного місяця. Історія даних для навчання моделі має складати не менше 36 попередніх місяців.

18. Інтерфейс користувача
Створення окремого інтерфейсу (програми або сайту) не передбачено. Взаємодія користувачів із результатами роботи алгоритму відбувається виключно через інтерфейс BI-платформи (графіки, тренди, таблиці порівняння плану та факту).

19. Ризики

Технічні збої або зміна структури таблиць у первинній базі даних маркетплейсу.

Різка зміна купівельної спроможності або поведінки споживачів через економічні кризи.

Поява агресивних нових конкурентів, що може різко знизити маржинальність продажів.

20. Примітки (для ведення проекту)
На першому етапі розробки (MVP) як базовий алгоритм буде використано модель Prophet, оскільки вона найкраще автоматично вловлює щорічну сезонність та дні свят на великих маркетплейсах.
